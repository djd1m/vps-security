# VPS Security — базовый хардинг Ubuntu VPS

Набор инструкций и скриптов для защиты Ubuntu VPS от наиболее распространённых атак: SSH-брутфорс, вход под root по паролю, открытые порты.

Создано на основе реального инцидента: сервер был полностью скомпрометирован через подбор пароля к SSH (`PermitRootLogin yes` + парольный вход). Атакующий установил C2-агентов, криптомайнер, прокси-ноду и создал бэкдор-аккаунт с sudo-правами.

## Когда что использовать

### Сценарий 1: «Хочу разобраться и сделать руками»

Используй **`01-hardening-guide-human.md`**.

Это инструкция для человека: каждая команда снабжена пояснением, что она делает и зачем. Подходит, если хочешь понять процесс и контролировать каждый шаг.

```bash
# Открой файл и следуй инструкциям
cat 01-hardening-guide-human.md
```

### Сценарий 2: «У меня есть Claude Code, пусть он сделает»

Используй **`02-hardening-prompt-claude-code.md`**.

Это промпт-задание для Claude Code. Он выполнит полный аудит сервера, покажет сводную таблицу проблем, применит исправления и проведёт верификацию. При этом он не отключит парольный вход, если нет SSH-ключа, и не будет удалять сервисы без подтверждения.

```bash
# Запусти Claude Code и передай ему промпт
claude "$(cat 02-hardening-prompt-claude-code.md)"
```

### Сценарий 3: «Новый сервер, хочу поставить Claude Code и захардить через него»

Используй **`03-install-claude-and-harden.sh`**.

Скрипт автоматически:
1. Проверит наличие SSH-ключа (без него не продолжит)
2. Установит Node.js 20
3. Установит Claude Code CLI
4. Запросит Anthropic API ключ
5. Запустит Claude Code с промптом из `02-hardening-prompt-claude-code.md`

```bash
# Предварительно: добавь SSH-ключ на сервер
# Затем:
chmod +x 03-install-claude-and-harden.sh
./03-install-claude-and-harden.sh
```

**Требования:** Ubuntu 22.04/24.04, root, SSH-ключ в `authorized_keys`, Anthropic API ключ.

### Сценарий 4: «Просто захардить, без Claude Code, в одну команду»

Используй **`04-harden-standalone.sh`**.

Полностью автономный скрипт. Не требует Claude Code, Node.js или API-ключей. Делает всё сам:
1. Аудит: SSH, брутфорс, fail2ban, фаервол, порты, подозрительные сервисы
2. Исправления: установка fail2ban, отключение парольного SSH, iptables-persistent
3. Итоговый отчёт в виде таблицы

Все действия логируются в `/var/log/vps-hardening-<дата>.log`.

```bash
# Предварительно: добавь SSH-ключ на сервер
# Затем:
chmod +x 04-harden-standalone.sh
./04-harden-standalone.sh
```

**Требования:** Ubuntu 22.04/24.04, root, SSH-ключ в `authorized_keys`.

## Перед запуском любого варианта

1. **Добавь свой SSH-ключ на сервер.** Без этого скрипты не отключат парольный вход (и правильно сделают — иначе потеряешь доступ).

   На Windows (PowerShell):
   ```powershell
   # Сгенерировать ключ (если нет)
   ssh-keygen -t ed25519

   # Скопировать на сервер
   type $env:USERPROFILE\.ssh\id_ed25519.pub | ssh root@<IP> "cat >> /root/.ssh/authorized_keys"

   # Исправить права (Windows)
   icacls "$env:USERPROFILE\.ssh\id_ed25519" /inheritance:r /grant:r "$($env:USERNAME):(R)"
   ```

2. **Проверь вход по ключу** — открой новую сессию SSH и убедись, что пароль не спрашивается.

3. **Не закрывай текущую SSH-сессию** до тех пор, пока не убедишься, что вход по ключу работает после хардинга.

## Что именно делают скрипты

| Действие | Зачем |
|---|---|
| `PermitRootLogin prohibit-password` | Root может войти только по ключу, пароль не принимается |
| `PasswordAuthentication no` | Парольный вход отключён для всех пользователей |
| Установка fail2ban | Автоматический бан IP после 3 неудачных попыток SSH на 24 часа |
| Установка iptables-persistent | Правила фаервола сохраняются после перезагрузки |

---

## Что скрипты НЕ делают

- Не включают UFW (может заблокировать нужные порты без подтверждения)
- Не удаляют сервисы (только сообщают о подозрительных)
- Не закрывают конкретные порты (зависит от того, что запущено на сервере)
- Не настраивают VPN/Tailscale
- Не настраивают централизованный сбор логов

Эти шаги рекомендуется выполнить вручную после базового хардинга.

---

## После хардинга рекомендуется

- [ ] Закрыть ненужные порты через iptables или UFW
- [ ] Настроить нестандартный порт SSH или ограничить доступ по IP
- [ ] Настроить мониторинг: новые systemd-сервисы, изменения authorized_keys, новые sudo-пользователи
- [ ] Настроить централизованный сбор логов (rsyslog remote или journald persistent)
- [ ] Контроль исходящих соединений (egress filtering)
